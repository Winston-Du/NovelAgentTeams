# 如何修正已生成章节并防止错误复现

> 适用于：novels_project CrewAI 小说创作系统
> 最后更新：2026-03-08

---

## 问题背景

你已经生成了多个章节（当前 chapter 1 ~ 16），发现其中存在写作方式、内容错误、结构等问题。你需要：

1. **修正已有章节**中的问题
2. **让 Agent 在后续创作中避免再犯同样错误**

本系统有 **四个可以利用的纠错通道**，按影响力从强到弱：

| 通道 | 作用范围 | 持久性 | 适合纠正的问题 |
|------|---------|--------|---------------|
| **反馈库** (`feedback/`) | 撰写员 + 校对读取 | 永久累积 | 写作手法、对话风格、Tell not Show 等 |
| **章节摘要卡** (`output/chapter_summaries/`) | 下一章的所有 Agent | 单章传递 | 剧情连续性错误、人物状态错误 |
| **写作样例库** (`samples/`) | 撰写员 + 校对检索 | 永久（向量库） | 文风、节奏、描写风格 |
| **Prompt 规则** (`DESIGN/PROMPTS/`) | 所有 Agent | 永久全局 | 系统性、反复出现的结构问题 |

下面按操作步骤详解。

---

## 第一步：诊断问题并分类

先把你发现的问题分类，不同类型走不同修正路径：

### 问题分类表

| 问题类型 | 举例 | 修正路径 |
|---------|------|---------|
| **A. 写作手法问题** | Tell not Show、流水账叙事、对话风格 OOC | → 反馈库 + 样例库 |
| **B. 内容/事实错误** | 人物名写错、律法条款矛盾、时间线冲突 | → 修正章节文件 + 修正摘要卡 |
| **C. 结构问题** | 铺垫不足直接高潮、爽点缺失、节奏拖沓 | → 反馈库 + Prompt 规则 |
| **D. 人物行为问题** | 角色 OOC、说话风格偏离、行为不合逻辑 | → 反馈库 + 人物卡库 |

---

## 第二步：注入反馈库 — 让 Agent "记住"你的修正

反馈库是最核心的纠错机制。撰写员和校对在**每次创作前都会查询**这个库。

### 2.1 反馈库的位置和格式

文件路径：`feedback/proofreading_feedback.yaml`

当前已有 3 条反馈记录。你可以直接编辑这个文件，手动添加反馈条目。

### 2.2 手动添加反馈

打开 `feedback/proofreading_feedback.yaml`，在 `feedback_history` 列表末尾追加条目。每条反馈的格式：

```yaml
- id: FB_MANUAL_001              # 手动反馈用 FB_MANUAL_ 前缀区分
  chapter_id: 16                  # 问题出现在哪一章
  issue_type: "对话风格偏离"       # 问题类型（见下表）
  character: "陆商曜"              # 涉及的人物（可选）
  original_text: "这里写有问题的原文片段（50字以内）"
  problem: "简要描述问题所在"
  fix_applied: "这里写你期望的正确写法"
  severity: "high"                # high / medium / low
  created_at: "2026-03-08T00:00:00"
```

**同时更新索引**：在 `feedback_by_type` 和 `feedback_by_character` 中添加对应的 ID。

### 2.3 常用 issue_type 值

系统内已预设的类型（Agent 会按这些关键词检索）：

| issue_type | 适用场景 |
|-----------|---------|
| `对话风格偏离` | 人物说话不符合卡库设定 |
| `Tell而非Show` | 直接描述情感而非用动作/对话表现 |
| `节奏问题` | 铺垫不足、拖沓、转折生硬 |
| `逻辑漏洞` | 因果链断裂、前后矛盾 |
| `重复结构` | "然后...接着...最后..." |
| `环境描写空洞` | 缺乏感官细节 |
| `爽点执行不力` | 铺垫不够、爆发无力 |
| `人物OOC` | 行为与性格设定不符 |
| `内容事实错误` | 人名/律法/设定与卡库矛盾 |
| `结构失衡` | 某部分篇幅过长或过短 |

你也可以自定义新的 issue_type，Agent 会按关键词匹配检索。

### 2.5 反馈如何被 Agent 消费

Agent 消费反馈的流程：

```
创作前：
  撰写员调用 retrieve_feedback() → 获取历史反馈 → 避免同类错误
  撰写员调用 get_common_mistakes() → 了解高频问题类型

校对时：
  校对调用 retrieve_feedback() → 重点检查历史高频问题
  校对发现新问题 → 调用 record_feedback() → 写入反馈库
```

**你手动添加的反馈会在下次运行时立即生效**，无需重启或重建。

## 第三步：修正章节摘要卡 — 阻断错误向下传播

摘要卡是章节间的**唯一数据桥梁**。如果某章的摘要卡包含错误信息，错误会传递给下一章的所有 Agent。

### 3.1 摘要卡的位置

```
output/chapter_summaries/chapter_{N}_summary.yaml
```

### 3.2 需要修正的场景

| 摘要卡中的错误 | 影响 | 修正方式 |
|--------------|------|---------|
| `character_status_updates` 中人物状态错误 | 下章人物设计师会基于错误状态生成状态卡 | 直接修改对应字段 |
| `foreshadowing_status` 中伏笔记录错误 | 下章总编可能遗忘或重复伏笔 | 修正伏笔列表 |
| `info_for_next_chapter` 中剧情连续性错误 | 下章总编会基于错误信息做大纲 | 修正 plot_continuity |
| `story_progress.key_events` 中事件描述错误 | 影响后续情节逻辑 | 修正事件列表 |

### 3.3 修正示例

假设第 16 章摘要卡中 `chapter_id` 写错了（当前写的是 2 而非 16）：

打开 `output/chapter_summaries/chapter_16_summary.yaml`，修正：

```yaml
chapter_id: 16    # 原来错误地写成了 2
```

假设人物关系描述不准确：

```yaml
character_status_updates:
  木九公:
    relationship_updates:
      与陆商曜: "信任度 20% -> 65%"    # 修正：原来写 50% 偏低
    status_change: "从悲观劝退到震惊认可，决定全力辅佐。"  # 修正描述
```

### 3.4 关键规则

- **只需修正最新一章的摘要卡**（即当前最后生成的那章），因为只有它会被传给下一章
- 如果要从中间某章重新生成，则需要修正那一章的摘要卡
- `continue_writing.py` 中的 `load_previous_summary()` 函数按 `chapter_{N-1}_summary.yaml` 加载上一章摘要

## 第四步：修正章节正文 — 直接改输出文件

### 4.1 文件位置

```
output/chapters/chapter_{N}_final.md     ← 最终版（Markdown 纯文本）
output/raw_outputs/chapter_{N}_raw.yaml  ← 原始 YAML 输出（含校对日志等结构化数据）
```

### 4.2 该改哪个文件

| 目的 | 改哪个 |
|-----|--------|
| 纯粹修正正文内容（错别字、对话、描写） | 改 `chapters/chapter_N_final.md` |
| 需要影响后续章节（如修正了关键剧情点） | 改 `chapter_summaries/chapter_N_summary.yaml` |
| 存档完整修正记录 | 也改 `raw_outputs/chapter_N_raw.yaml` 中的 `final_content` |

### 4.3 修正正文后的衔接

修正正文本身**不会自动影响后续章节**（因为 Agent 只读摘要卡，不读以前的正文）。如果你修正了关键剧情内容，**必须同步修正对应的摘要卡**。

## 第五步：添加「好样例」到样例库 — 教 Agent 正确的写法

样例库是撰写员和校对的**风格参考源**。目前只有 1 个样例文件：

```
samples/权谋章/opening_scene.md
```

### 5.1 添加新样例

在 `samples/` 目录下创建新的 `.md` 文件。建议按场景类型分目录：

```
samples/
├── 权谋章/
│   └── opening_scene.md          ← 已有
├── 战斗章/
│   └── iron_que_first_fight.md   ← 新增：铁阙首战（你从 chapter 16 中提取的好段落）
├── 情感章/
│   └── trust_building.md         ← 新增：信任建立场景
└── 纠错样例/
    └── show_not_tell_fixes.md    ← 新增：专门展示 Tell→Show 的修正对照
```

### 5.2 样例文件格式

参照已有的 `opening_scene.md` 格式：

```markdown
---
chapter_id: "纠错样例_001"
chapter_title: "Show not Tell 修正对照"
type: "纠错样例"
tags: ["Show not Tell", "动作描写", "环境渲染"]
focus: "用动作和环境替代直白情感描述"
scene_type: "修正对照"
word_count: 300
quality_score: 9.0
author_notes: "从第14-16章的修正中提取的好例子"
---

# Show not Tell 修正对照

## 错误写法 → 正确写法

### 案例1：表现愤怒
❌ "陆商曜感到愤怒"
✅ "陆商曜的指节在算盘上停了三息，随后拨珠的速度骤然加快，噼啪声像密集的雨点。"

### 案例2：表现紧张
❌ "气氛变得紧张起来"
✅ "灯火被窗缝里挤进来的风吹得一晃，所有人的影子在墙上突然变了形。"

### 案例3：表现决心
❌ "陆商曜下定了决心"
✅ "陆商曜把算盘合拢，扣入袖中，起身时脊背像一根打入地底的桩。"

---

## 技巧总结

1. **情感→动作**：用身体动作替代抽象情感
2. **氛围→环境**：用环境变化暗示氛围转变
3. **决定→行为**：用具体行为表现内心决定
```

### 5.3 重建向量库

添加新样例后，需要重建向量库让 Agent 能检索到：


## 第六步：更新 Prompt 规则 — 系统性防止问题复现

如果某类问题**反复出现**，说明 Prompt 层面的约束不够强。此时需要更新 Agent Prompt。

### 6.1 在哪里加规则

| 问题类型 | 改哪个 Prompt |
|---------|-------------|
| 大纲结构反复有问题 | `DESIGN/PROMPTS/chief_editor_prompt.md` |
| 人物状态卡反复不准 | `DESIGN/PROMPTS/character_designer_prompt.md` |
| 写作手法反复出错 | `DESIGN/PROMPTS/plot_writer_prompt.md` |
| 校对漏检问题 | `DESIGN/PROMPTS/proofreader_prompt.md` |

### 6.2 添加规则的方式

在对应 Prompt 文件的 `## ⚠️ 常见错误警示` 部分追加新条目。格式：

```markdown
### 错误N：[错误名称]
❌ [错误示例]
✅ [正确示例]
**原因**：[为什么这是错的]
**修正方法**：[怎么改]
```

### 6.3 示例：添加节奏控制规则

在 `plot_writer_prompt.md` 的常见错误部分追加：

```markdown
### 错误6：纯对话轰炸
❌ 连续 5 段以上纯对话，中间没有任何动作或环境描写
✅ 每 2-3 轮对话后穿插一段动作描写或环境变化
**原因**：长段纯对话会让节奏变得平面、单调
**修正方法**：在对话间隙添加"人物小动作"或"环境微变化"作为呼吸点
```

### 6.4 示例：添加事实一致性规则

在 `proofreader_prompt.md` 的校对检查清单中追加：

```markdown
### H. 事实一致性检查
- [ ] 引用的律法条款编号是否与前文一致？
- [ ] 人物的能力表现是否与卡库设定匹配？
- [ ] 地点名称、机构名称是否前后统一？
- [ ] 货币单位和数字是否合理？
```

## 第七步：更新人物卡库（如果问题涉及人物）

文件路径：`src/novels_project/config/character_base_cards.yaml`

### 7.1 需要更新的场景

| 问题 | 改动 |
|-----|------|
| 人物说话风格反复偏离 | 在 `avoid_patterns` 中添加具体的禁止项 |
| 人物行为逻辑不合理 | 调整 `core_personality` 或 `character_flaw` |
| 对话检查工具漏检 | 更新 `unique_speaking_style` 中的 `characteristics` |

### 7.2 示例

如果木九公的对话总是太直接，在他的人物卡中添加：

```yaml
木九公:
  unique_speaking_style:
    avoid_patterns:
      - "不要直接否定东家，应该用迂回方式提建议"        # 新增
      - "不要说超过两句话，每次发言最多两个短句"        # 新增
    characteristics:
      - "说话带方言韵味，用古雅词汇"
      - "习惯用比喻或典故代替直说"                     # 新增
      - "经常以咳嗽或叹气开头，再说正题"               # 新增
```

**注意**：修改人物卡后，`character_voice_checker.py` 中的规则检查也会自动更新（因为它直接读取这个 YAML 文件）。

## 操作流程总结

```
发现问题
    │
    ├─ 分类（写作手法 / 内容错误 / 结构 / 人物）
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ 立即生效（下次运行就能看到效果）                        │
│                                                     │
│ 1. 反馈库注入                                       │
│    → 编辑 feedback/proofreading_feedback.yaml        │
│    → 撰写员和校对创作前会读取                          │
│                                                     │
│ 2. 摘要卡修正                                       │
│    → 编辑 output/chapter_summaries/chapter_N.yaml    │
│    → 阻断错误传递给下一章                              │
│                                                     │
│ 3. 正文修正                                         │
│    → 编辑 output/chapters/chapter_N_final.md         │
│    → 纯修正存档，不影响 Agent                          │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│ 长期生效（防止错误反复出现）                            │
│                                                     │
│ 4. 样例库添加好样例                                   │
│    → 在 samples/ 下创建 .md 文件                      │
│    → 运行 python init_vectordb.py 重建向量库          │
│                                                     │
│ 5. Prompt 规则更新                                   │
│    → 编辑 DESIGN/PROMPTS/*.md                        │
│    → 在"常见错误警示"或"自检清单"中添加规则              │
│                                                     │
│ 6. 人物卡库更新                                      │
│    → 编辑 config/character_base_cards.yaml            │
│    → 加强 avoid_patterns 和 characteristics           │
└─────────────────────────────────────────────────────┘
```

## 快速参考：改动清单

每次修正问题时，按需勾选：

```
□ 1. 在 feedback/proofreading_feedback.yaml 中添加反馈条目
□ 2. 修正最新章节的 output/chapter_summaries/chapter_N_summary.yaml
□ 3. 修正正文 output/chapters/chapter_N_final.md（如需要）
□ 4. 在 samples/ 中添加好样例（如涉及文风问题）
     □ 4a. 运行 python init_vectordb.py 重建向量库
□ 5. 在 DESIGN/PROMPTS/ 中追加规则（如问题反复出现）
□ 6. 更新 config/character_base_cards.yaml（如涉及人物）
□ 7. 运行下一章验证修正是否生效
```
