# 混合模式人物管理指南

## 🎯 系统架构

```
人物管理系统（混合模式）
├── 主线人物（持续追踪）
│   ├── base_cards中定义
│   ├── character_type: "main"
│   └── 每章生成详细状态卡
│
├── 支线人物（需要时追踪）
│   ├── base_cards中定义
│   ├── character_type: "supporting"
│   └── 出现时生成中等详度状态卡
│
└── 临时人物（一次性）
    ├── 仅在章节大纲中指定
    ├── 不在base_cards中
    └── 仅生成当章简化状态卡
```

## 📝 实施指南

### 1. 主线人物设置

**在 `character_base_cards.yaml` 中定义**：

```yaml
s_tier:
  characters:
    陆商曜:
      name: "陆商曜"
      role: "主角"
      tier: "S_TIER"
      character_type: "main"  # ← 标记为主线
      core_personality: [...]
      # ... 完整的详细描述
```

**处理方式**：
- ✅ 人物设计师自动加载
- ✅ 每章生成详细状态卡
- ✅ 追踪跨章状态变化

---

### 2. 支线人物设置

**在 `character_base_cards.yaml` 中定义**：

```yaml
a_tier:
  characters:
    铁阙:
      name: "铁阙"
      role: "陆商曜的护卫"
      tier: "A_TIER"
      character_type: "supporting"  # ← 标记为支线
      core_personality: ["忠诚", "沉默", "执行力强"]
      core_motivation: "跟随陆商曜，保护他的安全"
      unique_speaking_style:
        tone: "极少说话，行动代替言语"
        # ... 简化的描述
```

**处理方式**：
- ✅ 人物设计师检查本章大纲是否涉及此人物
- ✅ 若本章出现：生成中等详度状态卡
- ✅ 若本章未出现：不生成状态卡
- ✅ 出现时追踪状态变化

**何时增加支线人物**：
- 经常出现，但非核心角色
- 需要在多个章节保持连贯性
- 或者你预计在后续章节还会出现

---

### 3. 临时人物设置

**在章节 `outline.md` 中指定**：

```markdown
## 第2章 - 市集风波

### 故事大纲
...

### 本章涉及人物

**主线/支线人物**：
- 陆商曜（主角，关键）- base_cards中定义
- 黑商周桓（反派，关键）- base_cards中定义
- 铁阙（护卫，重要）- base_cards中定义

**临时人物**（仅本章）：
- 赤眉匪头（临时反派）
  - 身份：盗匪头目，袭击货栈
  - 动机：抢劫，对陆商曜试探
  - 说话风格：粗野、蛮横、缺乏文化
  - 出场情景：第三幕，组织匪徒袭击

- 市集散户 × 3（临时配角）
  - 身份：普通商人，围观者
  - 动机：看热闹，评头论足
  - 说话风格：方言、议论纷纷
  - 出场情景：第二/三幕，作为背景人物
```

**处理方式**：
- ✅ 人物设计师读取outline中的"临时人物"部分
- ✅ 根据描述临时创建简化的角色卡
- ✅ 生成该章的简化状态卡（仅必要信息）
- ✅ 不做跨章追踪
- ✅ 后续章节如再出现，则在该章outline中重新定义

---

## 🔄 工作流程

### 创建一个新章节时

```
1. 编写 outline.md
   ├── 明确主故事线
   ├── 指定本章涉及的base_cards人物
   └── 定义新增的临时人物

2. 人物设计师 Agent 收到输入
   ├── 读取base_cards人物定义
   ├── 检查每个人物的character_type
   ├── 确定本章涉及的人物：
   │   ├── main → 生成详细状态卡
   │   ├── supporting → 检查是否涉及 → 生成中等状态卡或略过
   │   └── base_cards中无的 → 根据outline临时定义 → 生成简化状态卡
   └── 输出完整的chapter_character_states.yaml

3. 剧情撰写员使用状态卡进行创作

4. 完成后，将状态卡归档到 chapter_summaries/
```

## 📊 人物类型决策树

```
新人物出现？
│
├─ YES → 该人物需要在多个章节保持连贯吗？
│        │
│        ├─ YES → 该角色有深度背景和多面性吗？
│        │        │
│        │        ├─ YES → character_type: "main" (主线)
│        │        │        ➜ 加入 s_tier
│        │        │
│        │        └─ NO → character_type: "supporting" (支线)
│        │                ➜ 加入 a_tier/b_tier
│        │
│        └─ NO → 仅在本章出现，只需一次性角色吗?
│                │
│                └─ YES → 临时人物
│                         ➜ 仅在当章 outline 中定义
│
└─ NO → 什么都不做（可能你遗漏了一个角色)
```

## ✅ 最佳实践

### 何时添加到base_cards

✅ **添加为支线人物**：
- 将在2+个章节出现
- 需要持续的性格表现
- 与主线有关键互动

❌ **不添加**（改为临时人物）：
- 仅在1-2个场景出现
- 无跨章连贯性需求
- 角色作用有限

### 临时人物的简化程度

**三种简化程度**（根据出场重要性）：

1. **关键临时人物**（如本章反派）
   ```yaml
   赤眉匪头:
     core_personality: [...]
     motivation: ...
     speaking_style: ...
     # 相对完整
   ```

2. **普通临时人物**（如配角）
   ```yaml
   市集店主:
     role: "市集商人"
     personality_keywords: ["精明", "求生欲强"]
     # 简化，仅关键信息
   ```

3. **背景临时人物**（如群众）
   ```yaml
   路人:
     role: "围观群众"
     note: "方言，议论纷纷"
     # 极简
   ```

## 🚀 后续扩展

### 当人物库增长时

```
当前：3个S_tier + 2个A_tier = 5个人物

后续：
┌── S_TIER (5-8个)
│   └── 主线人物，持续追踪
├── A_TIER (10-15个)
│   └── 常出现的支线，需要追踪
├── B_TIER (20-30个)
```

### 迭代改进

1. **第一阶段**（当前MVP）
   - 3-5个主线人物
   - 5-10个支线人物
   - 每章1-3个临时人物

2. **第二阶段**
   - 扩充支线人物库
   - 建立人物关系图
   - 增加临时人物复用

3. **第三阶段**
   - 自动化检测章节中提及但未定义的人物
   - 提醒创建者补充定义
   - 智能推荐相似已有人物复用
